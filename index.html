The problem

Every new enterprise data-sharing engagement required the same manual sequence: review the customer's data schema, cross-reference it against anonymization capabilities, evaluate privacy risk using attacker-model results, draft an assessment, and route it for compliance sign-off. Each pass took most of a week — and most of that time was re-deriving context that had already been established in a prior, similar assessment.

The trust boundary

The design constraint that shaped everything else: the customer never gets access to internal systems. They hand over documents once — everything after that runs entirely inside internal infrastructure.

How it works

01

Context gathering

Pulls the customer's architecture docs and data specs from an internal Confluence space — never from the customer's own systems. (confluence_client.py)

02

Prior assessment search

The current use case is embedded and matched against a vector store of past assessments. A high similarity score lets the system reuse prior analysis instead of starting cold. (vector_store.py)

03

Attacker-model decision

Without a close precedent, a synthetic-data privacy-risk evaluation runs to quantify re-identification risk for this configuration. (attacker_model.py)

04

Draft generation

An LLM call produces a structured draft: threat model, quantified risk, configuration recommendation, and regulatory (GDPR) mapping — with risk level always computed deterministically from the score, never by the LLM. (draft_generator.py)

Two ways to run the same four steps

Worth being precise about a distinction that matters: a workflow and an agent are not the same thing, even when they call the same tools. This repo implements both, on purpose, to make the difference concrete rather than just claiming it.

Real tool-use loop, implemented directly against the wire API. The model chooses which tool to call and when — tested by call-counting whether it actually skips the expensive attacker-model run when a prior match exists.

MCP server

The same three read tools are also exposed over the Model Context Protocol (JSON-RPC over stdio), so any MCP-compatible client — Claude Desktop, another agent, a different application entirely — can call them without importing this repo's Python.

The one thing the LLM never decides

The riskiest failure mode for an LLM-in-the-loop compliance workflow isn't a bad draft — it's a confident, wrong risk classification that a reviewer rubber-stamps. Risk level (low/medium/high) is always computed deterministically from the attacker model's numeric score, never parsed from the LLM's output — enforced at the code level in the workflow, and at the tool-schema level in the agent (the submit_privacy_assessment tool doesn't even expose a risk_level field for the model to fill in). Both are tested explicitly, including a test where a fake LLM tries to smuggle in its own risk label and gets overridden anyway.
